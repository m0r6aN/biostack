# 1. Executive Summary

BioStack's backend is **far past "compound tracker."** It is a multi-project .NET 9 Clean Architecture system with a genuine evidence-governance spine, a deterministic offline knowledge-precompute worker, live Keon Collective deliberation, and a fail-closed Keon Runtime policy/receipt layer already wired into real endpoints. The hard, expensive scaffolding for a protocol intelligence engine is mostly built. What is missing is the **connective intelligence tissue** between the offline knowledge graph and the runtime reasoning surfaces, and the **evidentiary memory layer (Cortex)**, which is documented but not yet in code.

**What it already does well** `[OBSERVED, H]`
- Clean layer separation: `Domain` → `Application` → `Infrastructure` → `Api`, plus `Contracts`, `Cognition`/`Cognition.CollectiveAdapter`, and a standalone `KnowledgeWorker` precompute service.
- A real governance spine: `SpineEntry` append-only receipt ledger (`SpineRepository.cs`), Keon Runtime client with **fail-closed** policy/evidence gates (`KeonRuntimeClient.cs`), and a local pre-classifier (`PolicyGate.cs`, `DoctrineSanitizer`).
- A 101-field canonical `SubstanceRecord` source schema and a deterministic ingestion pipeline (load → validate → normalize → trust-gate → canonicalize) with a compound knowledge graph builder that already emits human-review findings (`CompoundGraphBuilder.cs`).
- Live multi-agent deliberation through Keon Collective, doctrinally locked to commentary-only (`CollectiveLiveOrchestrator.cs`, `StackReviewBoardService.cs`).
- Interaction intelligence with counterfactuals and swap recommendations (`InteractionIntelligenceService.cs`, `CounterfactualEngine.cs`, `SwapRecommendationEngine.cs`).
- Strong, layer-aligned test suite (Domain/Application/Api/KnowledgeWorker test projects, including governance unit tests).

**What is partial** `[OBSERVED/INFERRED, M]`
- Receipts exist but are **sparse and uniform** — almost everything is `"commentary-only"` with empty `EvidenceRefs` (see `ProtocolEndpoints.cs:256`). The receipt taxonomy is one class, not ten.
- The offline `CompoundGraph` and evidence packets are built by the worker but there is **no evidence that runtime services query the graph**; runtime interaction logic re-derives signals from `KnowledgeEntry` pipe-delimited string lists (`InteractionIntelligenceService.cs`).
- Knowledge is stored as denormalized string lists on `KnowledgeEntry` (`Pathways`, `AvoidWith`, etc. as `|`-joined columns) — fine for display, weak for graph reasoning.
- Transcript intake → candidate staging → review → promotion lifecycle is well-modeled, but the promotion **target is a single `KnowledgeEntry`**, not graph edges or evidence claims.

**What is missing** `[OBSERVED, H]`
- **Keon Cortex is not integrated in code** (zero `cortex` references in `src/`). It exists only in `docs/integration_ideas.md`. This is the single biggest gap between current state and the stated vision.
- No materialized intelligence artifacts at runtime (no substance intelligence profile table, no compatibility matrix store, no goal→pathway index served to the API).
- No vector/semantic retrieval; retrieval is pure relational + in-memory rule evaluation.
- No outcome-correlation / learning loop wired from `CheckIn` + `ProtocolRun` back into intelligence.
- No structured "why did BioStack say this" evidence-trail export.

**What is risky** `[OBSERVED/INFERRED, M]`
- Receipt `EvidenceRefs` are empty, so receipts currently prove *that* a decision happened, not *on what basis*. That undercuts the entire "receipts that matter" thesis.
- Hardcoded tenant/actor (`"biostack-public"`, `"biostack-system"`) in receipt issuance (`ProtocolEndpoints.cs:258-259`) — no per-user attribution in the audit chain.
- Runtime intelligence (interaction confidence, swaps) is generated by hand-tuned heuristics with magic numbers and is **not gated through the evidence/policy gate** before reaching the user in all paths. Verify every user-facing recommendation passes a gate.
- `KnowledgeEntry` carries dosing-shaped fields (`RecommendedDosage`, `WeeklyDosageSchedule`, `IncrementalEscalationSteps`) — these must never reach a user-facing surface without warning-first framing per canon §High-Risk.

**Verdict**

| Dimension | Maturity | Notes |
| --- | --- | --- |
| Backend architecture | **3.5 / 5 — Strong** | Clean Architecture, DI, tests, migrations all real. Needs contract/error-model consistency + graph-backed reads. |
| Protocol intelligence | **2.5 / 5 — Emerging** | Real interaction/counterfactual logic, rich offline graph, but runtime reasoning is heuristic and disconnected from the graph. |
| Data intelligence | **2 / 5 — Foundational** | Excellent source schema + precompute worker; no materialized runtime artifacts, no vector retrieval, no learning loop. |
| Keon integration readiness | **3 / 5 — Runtime live, Cortex absent** | Runtime + Collective live and doctrinally correct. Cortex unbuilt. Receipts thin. |
| **Recommended next phase** | **Lane B + Lane C + Lane G first** | Promote the knowledge model to graph/claim schema, materialize runtime intelligence artifacts, and thicken the receipt taxonomy with real evidence refs. |

# 2. Current Backend Architecture: How It Works

## 2.1 Project structure `[OBSERVED, H]`

```
backend/src/
  BioStack.Domain            # Entities, Enums, ValueObjects, Governance (SpineEntry)
  BioStack.Application        # Services (use-cases), Governance (PolicyGate, DoctrineSanitizer, Classification)
  BioStack.Infrastructure     # Persistence (EF Core DbContext + migrations), Repositories,
                              #   Keon (Runtime client/stub/options), Knowledge (IKnowledgeSource), Governance (SpineRepository)
  BioStack.Contracts          # Requests/ + Responses/ DTOs (the frontend-facing contract)
  BioStack.Cognition          # Collective API client + Stack Review Board service + deliberation translator
  BioStack.Cognition.CollectiveAdapter  # CognitiveDensity orchestrator + models
  BioStack.Api                # Minimal-API endpoints, Auth, Program.cs (composition root)
  BioStack.KnowledgeWorker    # Standalone offline ingestion/precompute pipeline + jobs + seeds + schemas
backend/tests/
  BioStack.Domain.Tests, BioStack.Application.Tests, BioStack.Api.Tests, BioStack.KnowledgeWorker.Tests
```

The `KnowledgeWorker` is a **separate executable** (`Program.cs`, `Dockerfile.knowledge-worker`) — the offline brain that compiles source material into promotable artifacts, kept out of the request path. This is the right shape for "super-intelligent preprocessing."

## 2.2 Layer boundaries `[OBSERVED, H]`
- `Domain` has no outward dependencies; entities are POCOs (`KnowledgeEntry.cs`, `Protocol.cs`, etc.).
- `Application` depends on `Domain` + `Infrastructure` interfaces (`IKnowledgeSource`, `IKeonRuntimeClient`, repositories).
- `Infrastructure` owns EF Core (`BioStackDbContext.cs`), the Keon HTTP clients, and the Spine.
- `Api` is the composition root; `Program.cs` wires ~60 services and maps ~22 endpoint groups.

## 2.3 API endpoint organization `[OBSERVED, H]`
Minimal APIs grouped per feature (`Program.cs:383-404`): Auth, Consent, Billing, Profile, Compound, CheckIn, Protocol, ProtocolPhase, ProtocolPortal, Timeline, Calculator, Knowledge, TrustLedger, StackReview, PolicyGate, Receipt, Analyze, Lead, Admin, (+ DevAuth in dev). Health checks at `/health` and `/health/keon`. OpenAPI + Scalar reference are mapped.

## 2.4 Dependency injection `[OBSERVED, H]`
Constructor injection throughout; `Program.cs` registers repositories (scoped), services (scoped), and stateless helpers (singleton — `BlendDecomposerService`, `ProtocolFingerprintService`, `ProtocolPortalBaseline`, `EvidenceGate`). HttpClients via `IHttpClientFactory` named clients (`keon-runtime`, `keon-collective`, `protocol-link-extractor`, `protocol-ocr`) — correctly avoids socket exhaustion.

## 2.5 Persistence model `[OBSERVED, H]`
EF Core, provider-resolved (`DatabaseProviderResolver.cs`, SQLite seen in check-constraints), code-first migrations (5 migrations from `RecoverBillingTierEnforcement` through `AddPromotionTargetToStagedTranscriptCandidateReviews`). Collections persisted as `|`-delimited strings via value converters (`BioStackDbContext.cs:345-380`). `TieredDosing` serialized as JSON. Spine + staged-candidate tables have real check constraints (`BioStackDbContext.cs:474-482`).

## 2.6 Entity/model relationships `[OBSERVED, H]`
`AppUser` → `PersonProfile` (multi-profile) → `CompoundRecord`, `CheckIn`, `Protocol`, `ProtocolRun`, `ProtocolPhase`, `TimelineEvent`. `Protocol` is **versioned** with parent/origin/evolved-from-run lineage (`Protocol.cs`, `BioStackDbContext.cs:189-221`) — strong foundation for outcome tracking. `ProtocolItem` snapshots compound fields (denormalized for historical integrity). Knowledge side: `KnowledgeEntry` (canonical), `CompoundInteractionHint` (curated pairwise), `InteractionFlag`. Governance side: `SpineEntry`, `KnowledgeSourceIntakeRequest`, `StagedTranscriptCandidateReviewEntity`.

## 2.7 Validation, error handling, auth `[OBSERVED/INFERRED, M]`
- Validation: request DTOs + service-level guards; `EvidenceGate` and `PolicyGate` are explicit fail-closed validators. No central FluentValidation layer observed `[INFERRED]`.
- Errors: `ProductErrorResponse` contract exists; `EvidenceGateViolationException : InvalidOperationException` deliberately maps to a 409 handler in `AdminEndpoints` (`EvidenceGate.cs:46-61`). Error mapping is **convention-based, not uniform** `[INFERRED, M]`.
- Auth: magic-link + JWT (`JwtTokenService`, `AuthEndpoints`, `HttpContextCurrentUserAccessor`), consent gating (`ConsentGate`, `RequireConsentFilter`), ownership guard (`OwnershipGuard`), feature/billing gating (`FeatureGate`, `BillingService`, Stripe). Roles via `UserRole` enum. Admin flows are role-gated `[INFERRED from AdminEndpoints + UserRole]`.

# 4. Current Feature Optimization

Each item: **Problem → Why → Fix → Difficulty → Priority → Tests → Receipt value**.

**O1. Receipts carry no evidence refs.**
- Problem: `IssueReceiptAsync` calls pass `EvidenceRefs: []` and `EffectStatus: "commentary-only"` uniformly (`ProtocolEndpoints.cs:262`, StackReview similar). `[OBSERVED, H]`
- Why: Receipts prove occurrence, not basis. Defeats "receipts that matter."
- Fix: Thread the evidence URIs (KnowledgeEntry ids, source refs, claim ids, graph finding ids) used to produce the decision into `EvidenceRefs`. Introduce a `ReceiptEvidenceBuilder` that intelligence services populate.
- Difficulty: M · Priority: **P0** · Tests: assert non-empty evidence refs per intelligence response · Receipts: this *is* the receipt fix.

**O2. Per-user/tenant attribution missing in receipts.**
- Problem: Hardcoded `"biostack-public"` / `"biostack-system"` (`ProtocolEndpoints.cs:258`). `[OBSERVED, H]`
- Fix: Pull tenant/actor from `ICurrentUserAccessor`; keep system actor only for genuinely system-initiated effects.
- Difficulty: S · Priority: **P0** · Tests: receipt actor == acting user · Receipts: enables personalization/user-observation receipt classes (§9).

**O3. Runtime intelligence ignores the precomputed CompoundGraph.**
- Problem: `CompoundGraphBuilder` produces a rich graph with synergy/conflict/risk findings, but `InteractionIntelligenceService` re-derives signals from `KnowledgeEntry` string lists at request time. `[OBSERVED, H]`
- Why: Two divergent truth sources; runtime logic can contradict reviewed graph.
- Fix: Persist the graph (Lane C) and have runtime read materialized edges/findings instead of recomputing.
- Difficulty: L · Priority: **P0** · Tests: runtime interaction == promoted graph edge · Receipts: intelligence-build + recommendation receipts gain real lineage.

**O4. Knowledge stored as `|`-delimited strings.**
- Problem: `Pathways`, `AvoidWith`, `PairsWellWith`, etc. are joined strings (`BioStackDbContext.cs:345-380`). `[OBSERVED, H]`
- Fix: Normalize into `Substance`, `Pathway`, `SubstancePathway`, `RelationshipEdge`, `EvidenceClaim` tables (§6.2). Keep the string columns as a denormalized read cache during migration.
- Difficulty: L · Priority: **P1** · Tests: round-trip + query-by-pathway · Receipts: enables claim-level evidence receipts.

**O5. Heuristic confidence with magic numbers.**
- Problem: Confidence constants (`0.72`, `0.64`, `0.59`, composite weights `18/14/12`) are inline (`InteractionIntelligenceService.cs:147-401`). `[OBSERVED, H]`
- Fix: Externalize to a versioned `ConfidenceModel` config; emit the model version into receipts (`ConfidenceProfile.CalibrationVersion` already exists in the Collective envelope — mirror it locally).
- Difficulty: M · Priority: **P1** · Tests: golden-file scoring · Receipts: confidence-model version becomes part of `policyHash`-style provenance.

**O6. Not every user-facing recommendation passes the policy/evidence gate.**
- Problem: Swap/counterfactual recommendation strings are generated and returned directly (`BuildRecommendation`, `InteractionIntelligenceService.cs:427`). `[INFERRED, M]`
- Fix: Route all generated guidance text through `PolicyGate.CheckAsync` before serialization; block/rewrite per decision.
- Difficulty: M · Priority: **P0 (safety)** · Tests: prohibited phrase → blocked · Receipts: safety receipts (§9).

**O7. Promotion target is a single KnowledgeEntry.**
- Problem: `TranscriptCandidatePromotionService` promotes to one canonical entry; canon expects evidence-tiered *relationships* and *claims*. `[OBSERVED, M]`
- Fix: Extend promotion to also create `EvidenceClaim` + `RelationshipEdge` artifacts.
- Difficulty: M · Priority: **P1** · Tests: promotion emits claims · Receipts: evidence-claim-created receipts.

**O8. Error model not uniform.**
- Problem: `ProductErrorResponse` exists but mapping is convention-based (exception subclass → status). `[INFERRED, M]`
- Fix: Central exception-mapping middleware → `ProductErrorResponse` with stable error codes.
- Difficulty: S · Priority: **P2** · Tests: each error code shape · Receipts: refusal/constraint receipts get stable codes.

**O9. Live transcript MCP client not wired.**
- Problem: `NullYouTubeTranscriptMcpClient` registered in DI (`Program.cs:283`). `[OBSERVED, H]`
- Fix: Provide a live `IYouTubeTranscriptMcpClient` behind config; keep null as fallback.
- Difficulty: S · Priority: **P2** · Tests: provider contract · Receipts: transcript-source-resolved receipts.

**O10. Health surface omits Collective/Cortex.**
- Fix: Add `/health/collective` and (future) `/health/cortex`. Difficulty: S · Priority: P2.

**F4. Support Foundation Analyzer** — Before higher-risk interventions, check support foundations (protein, hydration, sleep, resistance training, zone-2, electrolytes, liver/mito support). Educational "consider discussing X" only.
- Model: `SupportFoundationTemplate`. Collective: Support Foundation Council. Receipts: `intelligence.support-gap.surfaced`. Monetization: protocol readiness report. Priority: **P1**.

**F5. Synergy Graph** (read API over materialized edges) — serve `SynergizesWith`/`Complements`/`RedundantWith`/`ConflictsWith` edges with evidence tier + source authority mix. Priority: **P0** (it's already built offline; just persist+serve).

**F6. Contraindication & Caution Graph** — serve `AvoidWith`/`OpposesEffect`/`ContradictedBy` edges, warning-first. Priority: **P0**.

**F7. Evidence Confidence Engine** — central service returning evidence tier + classification confidence + uncertainty language for any claim. Wraps `EvidenceTier` + `SourceAuthorityMix`. Priority: **P1**.

**F8. Substitution Recommender** — already prototyped (`SwapRecommendationEngine`); upgrade to graph-backed + gated + receipted. Priority: **P1**.

**F9. Protocol Phase Generator** — derive descriptive phase labels from timeline events (non-instructional). Priority: **P2**.

**F10. Monitoring Plan Generator** — from `substance_requires_monitoring_with_biomarker` edges, produce a "markers to discuss with a professional" checklist. Monetization: paid monitoring plan. Priority: **P1**.

**F11. Outcome Correlation Engine** — correlate `CheckIn` deltas with protocol/phase changes ("what changed before this outcome"), never asserting causality (canon §Side-Effect Ambiguity). Data: `CheckIn`, `ProtocolRun`, `TimelineEvent`, drift snapshots. Cortex: `UserOutcomeObservationMemory`. Receipts: `user.outcome-observation.recorded`. Monetization: historical outcome analysis subscription. Priority: **P1**.

**F12. User Response Pattern Detector** — population-level aggregation of de-identified observations, marked as market/observational signal only. Priority: **P3** (privacy-heavy).

**F13. Compound Redundancy Detector** — runtime version of `InteractionType.Redundant`. Priority: **P2**.

**F14. Mechanism Collision Detector** — from `HasTarget`/`AffectsPathway` overlaps + opposing-effect edges (prototyped in `FindSameCategoryOpposingEffects`). Priority: **P2**.

**F15. Washout & Transition Planner** — descriptive washout/recovery phase context for *non-prescribed* substances only (canon §Phase). Priority: **P2**.

**F16. Diet & Training Support Planner** — templatized lifestyle support, educational. Priority: **P2**.

**F17. Cost-to-Value Optimizer** — combine `PricePaid` + evidence tier + redundancy to surface "high cost / low evidence / redundant" candidates for user reflection. Monetization: source-quality/cost report. Priority: **P2**.

**F18. Source Quality & Identity Risk Model** — runtime `SourceQuality` model from canon source classes (FDA-approved … gray-market … investigational). Warning-first. Receipts: `safety.identity-warning.issued`. Priority: **P1**.

**F19. Protocol Explainability Report** — "why did BioStack say this": assemble claims, edges, source refs, confidence, receipts into one bundle. This is the flagship Cortex consumer. Priority: **P1**.

# 6. Data Strategy: Data, Data, Data

## 6.1 Data sources `[PROPOSED]`
Peer-reviewed literature, trial registries (ClinicalTrials.gov), regulatory docs (FDA/EMA), PubMed abstracts, expert lectures/transcripts (already piped via intake), YouTube transcripts (intake exists), product labels, **user logs / check-ins / biomarkers / diet / exercise / sleep / wearables** (own entities), cost/source data (`PricePaid`/`Source`), adverse-event reports, and **community trend signals — clearly marked low-trust market signal** (already modeled as `CommunitySignal` with `CanonicalTruthStatus` in the graph). Each source carries an `authorityTier` (A1/A2/B1/B2…) per `SourceAuthorityMix`.

## 6.2 Data normalization — canonical schemas `[PROPOSED]`
Promote from string-lists to a normalized core (new EF entities):
- `Substance` (canonical identity; replaces denormalized `KnowledgeEntry` identity), `SubstanceAlias`, `Category`.
- `Mechanism`, `Pathway`, `Goal`, `SubstancePathway`, `SubstanceMechanism`, `GoalPathway`.
- `ProtocolPhaseRole` (substance↔phase semantics), `DoseConcept` / `RouteConcept` / `TimingConcept` (descriptive, non-prescriptive; flagged warning-first).
- `RelationshipEdge` (subject, object, type ∈ canon relationship families, phaseContext, goalContext, evidenceTier, confidence, sourceRefs, sourceAuthorityMix, safetyConcernLevel, productHandling, reviewStatus, userFacingExplanation) — **mirrors canon §Relationship Semantics exactly**.
- `ContraindicationSignal`, `InteractionSignal`, `SynergySignal`, `SupportRequirement`, `MonitoringMarker`.
- `EvidenceSource`, `EvidenceClaim` (claimType, statement, sourceRefs, fieldAuthorityRequired), `EvidenceTier`, `ConfidenceScore`, `ContradictionRecord`, `SafetyBoundary`.
- `UserOutcome`, `UserObservation`, `SourceQualityScore`.

The `KnowledgeWorker` already emits most of these shapes as JSON (`SubstanceRecord` 101 fields, evidence packets, relationship packets) — normalization is largely a *persistence projection*, not new modeling.

## 6.4 Retrieval strategy (hybrid) `[PROPOSED]`
- **Relational** for exact lookups (substance by canonical name/alias, user data).
- **Graph traversal** over `RelationshipEdge` for stack reasoning (synergy chains, opposing effects).
- **Materialized artifacts** for hot paths (compatibility matrix, goal index) — avoids recompute per request (fixes O3).
- **Vector retrieval** for semantic source/claim search + transcript dedup (new).
- **Structured filters:** evidence-tier filter, temporal filter (claim recency), user-profile filter (personalization overlay), **safety-boundary filter (always last, fail-closed)**.
- Rule of thumb: cheapest deterministic source first; vector only when no structured match; **always** terminate in the safety-boundary + evidence-gate filter before user-facing serialization.

## 6.5 Learning loop `[PROPOSED]`
Inputs: check-ins, protocol changes, reported outcomes, adherence, stops, substitutions, negative signals, missing data, care-team feedback, user corrections, admin review decisions. Only review-gated+ may change canonical artifacts; everything below stays labeled signal. Admin review decisions feed back as `ReviewDecisionIndex` (already exists in worker) → Cortex `ContradictionResolutionMemory`.

# 10. Safety, Compliance, and Trust Boundary `[OBSERVED strong + PROPOSED hardening]`

Current posture is genuinely good: canon defines observational-only boundaries; `PolicyGate` + `DoctrineSanitizer` block imperative/dosing/"is safe"/"cures" language; `EvidenceGate` fail-closes promotion lacking tier/citations/mechanism and scans for unsafe language; Collective is locked non-executable; Runtime gates fail closed.

Gaps + proposed gates:
- **Gate-all-outputs:** route *every* generated recommendation/summary string through `PolicyGate` (O6), not just promotion text.
- **High-risk category gate:** a `HighRiskCategoryGate` keyed on canon warning-first categories (SARMs, SERMs, investigational peptides, gray-market, compounded GLP-1s, prescription-only, banned-in-sport) that forces warning-first framing and blocks protocol-builder flows.
- **Investigational/peptide/SERM/SARM/GLP-1 handling:** never assert safety/efficacy; surface regulatory status + evidence limits + monitoring-discussion language only.
- **Unsafe / self-harm-adjacent requests:** explicit refusal path with a refusal receipt (audit value).
- **Care-team escalation prompts:** standardized "discuss with a qualified professional" inserts on high-risk or contradiction surfaces.
- **Auditability of refusals/constraints:** refused and rewritten outputs get receipts (safety family) so the constraint itself is provable.
- **Runtime enforcement point:** Keon Runtime is the enforcement boundary for policy/evidence; BioStack's local pre-classifier is a fast-path only, never the sole gate.
